Output Format & Tool Validation Protocol
Repeated XML format errors (missing tool blocks, unclosed tags), JSON output after tool blocks, hallucinated tools, or invalid tool parameters
Lesson/Fix: Strictly adhere to the output format: [Reasoning Text] immediately followed by exactly one valid &lt;tool&gt; block. No text or JSON may follow the tool block. Before generating any tool call, validate the tool name exists and parameters match the schema. If a format error occurs, abort and output a corrected response immediately. Do not repeat failed tool calls in consecutive turns. Only call tools that are verified to exist and match the schema before execution.

Execute After Planning - No Planning-Only Mode
Task completed in "Plan" phase without actual file creation or command execution. Files never created, npm init never executed, task marked FAILED despite "no tool errors" being reported
Lesson/Fix: Planning and execution are separate phases. After planning is complete, immediately execute ALL required commands before considering planning done. File creation and command execution are deliverables - they must be created, not just planned. Call the appropriate execution tools (write_file, etc.) and verify their success before finishing the task. Do NOT declare files as created in Plan or Summary stages before actual execution.

Verify Tool Results Before Proceeding
Agent proceeds to next step without confirming previous tool call succeeded. Tests fail, file verification incomplete, task marked incomplete
Lesson/Fix: After every tool call (list_dir, write_file, run_command, etc.), verify the result before proceeding. Use read_file to confirm file creation, check run_command output for success, or verify directory structure with list_dir. File creation is complete only after verification. Script execution is complete only after output check. Never proceed to the next step without confirming the previous step succeeded.

Critical Tool Failure Handling
Multiple tool failures (list_dir fails 2+ times, workspace verification failures). Agent retries same failing tool instead of stopping. Workspace assumed accessible when it is not
Lesson/Fix: When any critical verification tool (list_dir for workspace verification, read_file for file status, etc.) fails 2+ times, immediately STOP execution and report the failure with clear error details to the meta-agent. Do not retry the same failing tool call in consecutive turns. The agent should not assume the workspace is accessible if verification fails. Report the error clearly with the specific failure message to enable meta-agent recovery. Never proceed to the next step if a critical tool failure occurred.

Web Artifact Execution Verification
Agent generated web files but failed to execute them in browser. Task marked FAILED due to sandbox limitations instead of recognizing file creation as deliverable
Lesson/Fix: For web artifacts (HTML/JS/CSS), file content verification is insufficient. Attempt to launch the file in a browser or local server (xdg-open/start/open). If execution isn't possible due to sandbox limitations, do not mark the task FAILED. The file creation itself is the deliverable. Call finish_task with a verdict like "SUCCESS (sandbox limitation)" or "SUCCESS - file ready for browser testing" and document the limitation in the summary. Sandbox limitations do not make a task failed.

Verification Overhead on Simple Tasks
Agent called verification tools (list_dir) for every step before execution, even for standard paths. When these failed, task execution never happened.
Lesson/Fix: Do not call verification tools (like list_dir) for every task step. Only use them if the workspace structure is explicitly unknown or paths are complex. Prioritize executing the required commands (write_file, run_command) immediately after planning. If a tool fails, do not assume the workspace is broken and proceed to the next step unless verification is the only barrier.

Tool Failure vs. Task Failure
Agent stopped execution upon tool failure but reported "Task Failed" instead of "Tool Failure Reported". This prevents the Meta-Agent from knowing the tool is the root cause.
Lesson/Fix: When a critical tool (list_dir, read_file, etc.) fails 2+ times, the agent must explicitly stop execution and report the *tool failure* to the Meta-Agent (e.g., "Tool Failure: list_dir failed 2 times. Workspace may be inaccessible. Aborting task execution"). Do not assume the task is failed solely due to tool issues; the Meta-Agent may need to reset or debug the environment first.
