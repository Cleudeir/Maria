## DYNAMIC GUIDELINES & LESSONS LEARNED:
### XML Protocol & Tool Call Integrity
- Typical Error: Repeated XML format errors (missing tool blocks, unclosed tags) causing parsing failures; redundant or looping tool calls (e.g., rewriting the same file without logic change); tool hallucinations.
- Resolution: Strictly adhere to one XML format: [Reasoning Text] followed immediately by exactly one valid &lt;tool name='...'&gt;...&lt;/tool&gt; block. No text or JSON after the tool block. Before generating any tool call, validate the tool name exists and parameters match the schema. Before executing any write command, check file status to avoid redundant writes. If a format error occurs, abort and output a corrected response immediately. Do not repeat failed tool calls in consecutive turns.

### File System Safety & Workspace Discovery
- Typical Error: Errno 21 (Is a directory), using absolute paths, ignoring codebase structure, writing to non-existent directories, or looping on successful file writes.
- Resolution: Before writing any file, always use &lt;tool&gt;list_dir&lt;/tool&gt; to explore the workspace and confirm the target path exists as a file (not a directory). Use relative paths from the workspace root. If the target directory doesn't exist, create it first. If a file already exists and matches the plan, do not rewrite it. Always verify the file was successfully written (e.g., read_file) before proceeding to the next step.

### Workflow Discipline & Task Definition
- Typical Error: Tests failing unexpectedly, skipping TDD for complex logic, conflating task deliverable with execution verification, or finishing without verifying output.
- Resolution: Maintain the full TDD cycle (Write Failing Test -> Verify -> Implement -> Verify) for Python tasks; simplify for trivial file creation but keep verification rigorous. Proactively execute plans immediately if the workspace is empty and clear. Distinguish the task deliverable from execution: for file tasks, file creation is complete; for script tasks, execution is complete. Handle Python cache issues proactively. Even in simplified workflows, explicitly verify that code was executed and produced expected output, not just that the file content was written.

### Web Artifacts: Runtime Verification & Environment Handling
- Typical Error: Agent generated web files but failed to execute them in a browser; task marked FAILED due to sandbox limitations (no browser/server); port conflicts; self-assessment mismatching system verdict.
- Resolution: For web artifacts (HTML/JS/CSS), file content verification is insufficient. Attempt to launch the file in a browser (xdg-open/start/open) or a local server. If direct execution isn't possible, try 2-3 methods before giving up. If all attempts fail due to environment limitations, do not mark the task FAILED. The file creation itself is the deliverable. Call finish_task with a verdict like "SUCCESS (sandbox limitation)" or "SUCCESS - file ready for browser testing" and document the limitation in the summary. Do not continue trying multiple browser launch methods if the file creation is successful.
