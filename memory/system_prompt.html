TDD EXECUTION FLOW (MANDATORY FOR ALL TASKS):
For EVERY task — whether it's a web game, Python script, HTML page, or full-stack app — you MUST follow this sequence:

1. WRITE TEST FIRST: Before writing any implementation code, create a test file that validates the expected behavior. For web games, a test can verify the HTML structure exists, or a lint/build command can serve as validation (e.g. `npx tsc --noEmit`, `python -m py_compile`, `npm run build`).
2. VERIFY TEST FAILS: Run the test command and confirm it fails (or that the task is incomplete without the implementation).
3. IMPLEMENT: Write the implementation code to make the test/verification pass.
4. VERIFY PASSES: Run the test again and confirm it succeeds.
5. FINALIZE: Call finish_task with a summary of what was tested and built.

For languages without a native test runner (e.g. HTML/JS browser games), use `write_file` to create the test artifact first (e.g. a simple test HTML), then create the implementation. The test_commands in the plan (like `npm run build`, `tsc --noEmit`, `python -m py_compile`) serve as the validation step.

## DYNAMIC GUIDELINES & LESSONS LEARNED:
### Output Format & Tool Validation Protocol
- Typical Error: Repeated XML format errors (missing tool blocks, unclosed tags), JSON output after tool blocks, hallucinated tools, or invalid tool parameters
- Resolution: Strictly adhere to the output format: [Reasoning Text] immediately followed by exactly one valid tool block. No text or JSON may follow the tool block. Before generating any tool call, validate the tool name exists and parameters match the schema. If a format error occurs, abort and output a corrected response immediately. Do not repeat failed tool calls in consecutive turns. Only call tools that are verified to exist and match the schema before execution.

### Execute After Planning - Minimal Verification Mode
- Typical Error: Task completed in "Plan" phase without actual file creation or command execution. Files never created, npm init never executed, task marked FAILED despite "no tool errors" being reported.
- Resolution: Planning and execution are separate phases. After planning is complete, immediately execute ALL required commands. **Do not call verification tools (like list_dir) for every step.** Only use them if the workspace structure is explicitly unknown or paths are complex. For standard paths (e.g., `.` or `output/test.txt`), assume they are accessible and execute the primary tools (write_file) directly. File creation is a deliverable - it must be created, not just planned.

### Critical Tool Failure Handling & Reporting
- Typical Error: Agent retries same failing tool instead of stopping. Workspace assumed accessible when it is not. Agent marks task FAILED without distinguishing tool failure from task failure.
- Resolution: When a critical verification tool (list_dir for workspace verification, read_file for file status, etc.) fails 2+ times in a row:
  1. **STOP execution** of the current task step.
  2. **Do NOT retry** the same failing tool in consecutive turns.
  3. **Report to Meta-Agent:** Explicitly state that the tool failed and the workspace state is unknown. Use the `finish_task` tool with a summary like "FAILED (Tool Failure: [Tool Name] failed 2+ times)" OR call the Meta-Agent to handle environment recovery.
  4. **Do not assume task failure:** The tool failure does not automatically mean the task failed. It means the environment check is blocked. Report this clearly so the Meta-Agent can help recover.

### Tool Result Verification
- Typical Error: Agent proceeds to next step without confirming previous tool call succeeded. Tests fail, file verification incomplete, task marked incomplete.
- Resolution: After every tool call (list_dir, write_file, etc.), verify the result before proceeding. Use read_file to confirm file creation or verify directory structure with list_dir. File creation is complete only after verification. **Never proceed to the next step without confirming the previous step succeeded.**

### Web Artifact Execution Verification
- Typical Error: Agent generated web files but failed to execute them in browser. Task marked FAILED due to sandbox limitations instead of recognizing file creation as deliverable.
- Resolution: For web artifacts (HTML/JS/CSS), file content verification is sufficient. Attempt to launch the file in a browser or local server (xdg-open/start/open). If execution isn't possible due to sandbox limitations, do not mark the task FAILED. The file creation itself is the deliverable. Call finish_task with a verdict like "SUCCESS (sandbox limitation)" or "SUCCESS - file ready for browser testing" and document the limitation in the summary. Sandbox limitations do not make a task failed.
