## DYNAMIC GUIDELINES & LESSONS LEARNED:
### JSON Format Compliance & Recovery
- Typical Error: Format error - response could not be parsed into a valid tool call JSON.
- Resolution: JSON format is a HARD CONSTRAINT. Every output MUST contain reasoning followed by exactly one JSON object: {"tool": "tool_name", "args": {...}}. If a format error occurs, immediately output a corrected response with proper JSON format. Never output XML tags.

### TDD Workflow & Execution Discipline
- Typical Error: Tests fail unexpectedly (e.g., ModuleNotFoundError), implementation occurs before verifying test failure, or TDD workflow is skipped when no errors occur.
- Resolution: Adhere strictly to the TDD cycle regardless of task difficulty or initial success. 1) Write a failing test first (including frontend tests for canvas, events, game logic, etc.). 2) Verify the failure. 3) Implement code only to satisfy the failing test. 4) Verify success. Process over perfection: Workflow compliance is more important than avoiding immediate errors. Do not skip TDD for straightforward tasks.

### File System Safety & Context Awareness
- Typical Error: Errno 21 Is a directory when writing files, errors using absolute paths, or ignoring existing codebase structure.
- Resolution: Before writing files, ALWAYS verify the target path exists as a file (not a directory) using list_dir. Use relative paths from the workspace root (e.g., 'index.html') instead of absolute paths. Explore the workspace structure first to ensure new files fit within the correct directory layout.

### Task Completion & finish_task Usage
- Typical Error: Task step loops indefinitely without calling finish_task, or marks complete without verifying code was written.
- Resolution: When you have completed ALL the work for the current step (written files, run tests, verified results), IMMEDIATELY call {"tool": "finish_task", "args": {"summary": "what you did"}}. Do NOT continue querying the workspace or re-reading files after completing the work. The finish_task tool is how you signal step completion to the system. If you keep making tool calls without finishing, you are stuck in a loop - call finish_task.

### Avoiding Repetitive Loops
- Typical Error: Model repeats the same tool call (e.g., reading the same file multiple times) without making progress or calling finish_task.
- Resolution: After 2-3 tool calls for the same sub-task, evaluate whether the step is complete. If you have already written the necessary code and verified it works, call finish_task immediately. Do not re-read files you just wrote. Do not re-run tests that already passed. Progress means moving toward finish_task, not gathering more information.

### Handling Empty Workspaces with Concrete Implementation
- Typical Error: Agent completed task with empty workspace and placeholder text in execution steps. No code was generated despite having a complete implementation plan.
- Resolution: When you recognize an empty workspace or placeholder execution steps, immediately start implementation without waiting for user input. For game development tasks, begin writing core files (game.py) directly. Use list_dir to verify workspace, then write_file for implementation. Only finish_task after confirming files exist and tests pass.

### Proactive Code Generation vs. Asking for Input
- Typical Error: Agent asked for user input when workspace was empty and had a complete plan, rather than executing the plan directly.
- Resolution: When you have a complete implementation plan and the workspace is empty, execute the plan directly. Do not ask the user for input. Generate files: game.py (implementation), test_game.py (tests), requirements.txt (dependencies). Only ask for user input if the plan itself is incomplete or unclear.
