## DYNAMIC GUIDELINES & LESSONS LEARNED:
### XML Protocol Compliance
- Typical Error: Format errors: Missing <thought> tags, unclosed <tool> tags, or tool calls without preceding reasoning.
- Resolution: Strictly enforce the XML structure for all tool interactions: 1. Begin every tool call with a <thought>...</thought> block containing your reasoning. 2. Wrap the tool call in <tool name="...">...</tool> tags with proper opening and closing. 3. Never output a tool call directly without the thought tag first. Ensure all tags are properly closed to prevent parser errors.

### TDD Workflow Discipline
- Typical Error: Tests fail unexpectedly (e.g., ModuleNotFoundError), or implementation occurs before verifying test failure.
- Resolution: Adhere strictly to the Test-Driven Development cycle: 1. Write a failing test first. 2. Verify the failure (confirm the error matches expectations). 3. Implement code only to satisfy the failing test. 4. Verify success. Do not implement code that does not match the specific failure state of the test.
- CRITICAL: Follow TDD workflow regardless of whether errors occur. Even when a task completes successfully, write failing tests first to ensure code quality, catch edge cases, and maintain consistent development patterns.

### File System Safety & Path Validation
- Typical Error: Errno 21 Is a directory when writing files, or errors using absolute paths.
- Resolution: Before calling write_file, ALWAYS verify the target path exists as a file (not a directory) using list_dir or os.path.isfile. Use relative paths from the workspace root (e.g., 'index.html') instead of absolute paths or the workspace root itself. Add a validation step to catch path errors early.

### Workspace Context Awareness
- Typical Error: Attempting to write in non-existent locations or ignoring existing codebase structure.
- Resolution: Before starting any implementation, explore the workspace structure using list_dir to understand the existing codebase layout. This ensures new files fit within the correct directory structure and prevents path errors. Always verify the workspace root is a directory, not a file, before attempting operations.

### TDD for Frontend Tasks
- Typical Error: Frontend tasks implemented without tests, despite TDD being required.
- Resolution: Apply TDD to frontend tasks by: 1. Writing tests that verify canvas rendering and game state 2. Testing event handling for keyboard inputs 3. Verifying particle system behavior 4. Testing score tracking and localStorage persistence. Use run_command to execute browser-based tests or write unit tests that can be run in the workspace.
