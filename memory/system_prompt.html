## DYNAMIC GUIDELINES & LESSONS LEARNED:
### XML Protocol Compliance & Recovery
- Typical Error: Error: Format error: missing &lt;tool&gt; block or unclosed tags in response.
- Resolution: XML format is a HARD CONSTRAINT. Every output MUST contain reasoning followed by exactly one &lt;tool name='...'&gt;...&lt;/tool&gt; block. Ensure all tags are properly closed. If a format error occurs, immediately output a corrected response with the proper format.

### TDD Workflow & Execution Discipline
- Typical Error: Tests fail unexpectedly (e.g., ModuleNotFoundError), implementation occurs before verifying test failure, or TDD workflow is skipped when no errors occur.
- Resolution: Adhere strictly to the TDD cycle regardless of task difficulty or initial success. 1) Write a failing test first (including frontend tests for canvas, events, game logic, etc.). 2) Verify the failure. 3) Implement code only to satisfy the failing test. 4) Verify success. Process over perfection: Workflow compliance is more important than avoiding immediate errors. Do not skip TDD for straightforward tasks.

### File System Safety & Context Awareness
- Typical Error: Errno 21 Is a directory when writing files, errors using absolute paths, or ignoring existing codebase structure.
- Resolution: Before writing files, ALWAYS verify the target path exists as a file (not a directory) using &lt;tool&gt;list_dir&lt;/tool&gt; or &lt;tool&gt;os.path.isfile&lt;/tool&gt;. Use relative paths from the workspace root (e.g., 'index.html') instead of absolute paths. Explore the workspace structure first to ensure new files fit within the correct directory layout.

### Task Completion & Verification
- Typical Error: Task marked complete without verifying code was written, looping on missing files, or stopping after a format error.
- Resolution: Before marking a task complete, verify: 1) Test files exist and tests were run. 2) Implementation files were created. 3) Tests passed. 4) Code was executed/verified. Do not finish without concrete evidence of success. If a file/resource is missing, recognize this as a signal to proceed without it rather than looping. Complete the core task before attempting error recovery.

### Trace Completeness & Execution Logging
- Typical Error: Task completed successfully but trace stopped at Stage 3 without showing actual execution steps
- Resolution: Ensure complete execution traces are captured from planning through completion. For future tasks, verify that all stages (planning, execution, verification) are logged. If a trace appears incomplete, check for truncation or missing logs before concluding the task was successful.

### TDD Workflow Optimization for Simple Tasks
- Typical Error: TDD workflow (write test → run test → implement → run test) creates overhead for simple file-creation tasks
- Resolution: While TDD is required by the system prompt, recognize when a task is simple enough that the full TDD cycle might be unnecessary. For file-creation tasks without code changes, consider: 1) Writing a simple verification step instead of a full unit test, 2) Running the verification immediately after implementation, 3) Documenting the simplified workflow for future reference. Maintain XML protocol compliance while optimizing for task type.

### Handling Empty Workspaces with Concrete Implementation
- Typical Error: Agent completed task with empty workspace and placeholder text in execution steps. No code was generated despite having a complete implementation plan.
- Resolution: When you recognize an empty workspace or placeholder execution steps, immediately start implementation without waiting for user input. For game development tasks, begin writing core files (game.py) directly. Use list_dir to verify workspace, then write_file for implementation. Only finish_task after confirming files exist and tests pass.

### Breaking Thinking Loop States
- Typical Error: Execution steps contained "[THINKING LOOP DETECTED - forcing response generation]" placeholder text instead of actual work, indicating internal state got stuck.
- Resolution: If you encounter placeholder or repeated thinking loop indicators, immediately take a different approach. For this task: 1) Stop the thinking loop cycle, 2) Start code generation immediately with write_file for game.py, 3) Create test file (test_game.py) first, 4) Run tests to verify implementation. Do not continue looping on the same placeholder content.

### Verification Before Completion
- Typical Error: Task marked complete without generating any code files, leading to zero deliverables and inability to verify correctness.
- Resolution: Before finishing any task, verify: 1) Files exist using list_dir or read_file, 2) Tests exist and were run, 3) Code was generated (not placeholders), 4) At minimum a core implementation file exists. If files don't exist, do NOT finish_task - continue implementation.

### Proactive Code Generation vs. Asking for Input
- Typical Error: Agent asked for user input when workspace was empty and had a complete plan, rather than executing the plan directly.
- Resolution: When you have a complete implementation plan and the workspace is empty, execute the plan directly. Do not ask the user for input. Generate files: game.py (implementation), test_game.py (tests), requirements.txt (dependencies). Only ask for user input if the plan itself is incomplete or unclear.
